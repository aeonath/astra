# Changelog — 2026-03-01 01:00

## Database Durability & Deployment Safety

Overhauled the database layer to guarantee that normal code updates, redeploys, or git pulls can never wipe or recreate the database. Data durability is now the top priority.

### What changed

- **No auto-creation on startup:** Astra no longer auto-creates or auto-initializes the database when the app starts. If the database file is missing, the app fails loudly with clear instructions.

- **Explicit initialization:** New `npm run db:init` command creates a fresh database at the configured path. Refuses to overwrite an existing database.

- **Forward-only migrations:** New `npm run db:migrate` command applies pending versioned migrations. Migration files live in `src/db/migrations/` and export `{ version, name, up(db) }`. Tracked in a `schema_migrations` table. No down migrations, no destructive operations.

- **Startup verification:** On startup, Astra verifies:
  - `DB_PATH` env var is set and is an absolute path
  - The database file exists
  - The `schema_migrations` table is present
  - No pending migrations remain
  - If any check fails, the app refuses to start with a clear error message.

- **Database path externalized:** `DB_PATH` must be an absolute path outside the source tree (e.g. `/var/lib/astra/astra.db`). Session database directory is also configurable via `SESSION_DB_DIR`.

- **Legacy migration bridge:** Running `npm run db:migrate` on existing databases automatically detects the old `_migrations` table and bridges it to the new `schema_migrations` system.
